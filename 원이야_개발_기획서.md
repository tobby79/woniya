# 원이야(Woniya) 개발 기획서 (v4)

> 동네 어린이집·유치원을 위한 모바일 홈페이지 & 검색 플랫폼
> v3 대비 변경: 페이지 구조 전략(18번) 개편, 버스 노선 페이지(19번) 신설, 온보딩 수집 폼(20번) 신설, 원장 어드민·대시보드(21번) 신설, 데이터 모델 보강, 마일스톤 업데이트.

---

## 1. 문서 목적

- 프로토타입(단일 HTML 데모)에서 실제 서비스로 넘어가기 위한 개발 범위·아키텍처·일정을 정의한다.
- 학부모(B2C)와 원장(B2B) 동선을 하나의 제품으로 구현하는 기준을 잡는다.
- 원장을 점수로 줄세우지 않는 신뢰 신호 설계와 개인정보 안전 설계를 처음부터 코드에 반영한다.

---

## 2. 제품 개요

- 한 줄 정의: 학부모는 동네 원을 검색·비교하고, 원은 제작비 없이 세련된 모바일 홈페이지와 운영 도구를 갖는 SaaS.
- 두 사용자:
  - 학부모(B2C): 원 검색·비교 → 원 홈페이지 열람 → 상담·입소 신청 → 초대장형 통신문 수신·응답 → 반 미니홈피에서 우리 반 일상 확인.
  - 원장/교사(B2B): 홈페이지 자동 완성 → 정원·상담 관리 → 통신문·동의서(초대장) 발송 → 공식 문의 응대 → 원장 커뮤니티 → 담임 교사가 반 미니홈피 운영.
- 현재 상태: 템플릿 4개(sunshine/forest/carnival/gallery) 운영 중. Supabase DB 연동, 입학안내·오시는 길·메뉴 시스템 구현 완료.
- 핵심 자산: 약 200명 원장 네트워크(소개 기반 확산), 모바일 인터랙티브 페이지 제작 역량.
- 차별점: 기능이 아니라 "감성 있는 인터랙티브 페이지". 흔한 공문·알림을 초대장 같은 작품으로 만든다.

## 3. 개발 목표 & MVP 범위

### MVP 포함 (1차 출시)

- 원 모바일 홈페이지 빌더: DB 기반 렌더, 원장이 사진·문구만 넣으면 완성.
- 초기 온보딩 세팅: 원장이 온보딩 수집 폼으로 정보 제출 → 드림파파가 1회 세팅 → 원장에게 콘솔 인도. 상세는 20번 참조.
- 원 정보 콘텐츠: 입학안내·오시는 길 필수, 소개 영상·원 철학·교육과정·버스 노선 선택(원장 ON/OFF). 상세는 17번·19번 참조.
- 원장 인증·콘솔(기본): 로그인, 원 정보·사진·정원 편집. 직관적 어드민 + 대시보드 포함. 상세는 21번 참조.
- 정원 상태 + 상담·대기 신청(전환 다리): 학부모 신청 → 원장 접수.
- B2C 검색·비교: 위치 기반 검색, 결과 카드(빈자리·사실 배지), 원 홈 연결. (평점 없음)
- 검증 배지 + 적합도 태그 + FAQ 챗봇(룰 기반): 정보 선제 공개로 문의·민원 절감.
- 초대장형 통신문·동의서 + 열람/응답 기록: 발송·수신·한 탭 응답·타임스탬프 기록.
- 공식 문의 창구(악성 민원 방어 1차): 접수번호·시각 기록, 원장 응대.

### 2차 이후 (확장)

- 응대 템플릿 + AI 답변 초안(공식 문의 고도화).
- 원장 커뮤니티(비공개 피드·공동구매).
- 결제·구독 자동화(Toss 정기결제), 업셀(프리미엄 템플릿·독자 도메인·앨범 커머스).
- 지도 핀 검색 고도화, 챗봇 AI 전환.
- 반 미니홈피: 담임 선생님이 운영하는 반 전용 비공개 공간. 해당 반 학부모만 입장 가능. (16번 참조)

### 범위 밖 (당분간 개발하지 않음)

- 아이 개인정보 카드를 부모별로 1:1 공유하는 기능. (키즈노트 영역 충돌 + 민감정보 책임 급상승)
- 공개 별점·후기(리뷰). (원장 압박·리뷰 폭격 위험. 5번 원칙 참조)
- 회계 장부·정산 기능. (전문 서비스 영역 충돌, 포지셔닝 흐림. 추후 업셀로 재검토)
- 버스 실시간 GPS 추적. (기사 폰 앱 필요, 개인정보 이슈, 키즈노트 영역 충돌. 19번 참조)

## 4. 핵심 사용자 시나리오

### 학부모

- 동네 검색 → 원 비교(사진·빈자리·사실 배지·적합도 태그) → 원 홈 열람(하루 스토리) → 메뉴에서 입학안내·오시는 길·버스 노선 모달로 확인 → FAQ 챗봇 → 상담·대기 신청 → 초대장형 통신문 수신·한 탭 응답.

### 원장

- 온보딩 수집 폼 제출 → 드림파파 1회 세팅 → 콘솔 인도 → 원 정보·사진·정원 직접 편집 → 상담·문의 확인·응대 → 통신문·동의서 작성·발송 → 대시보드에서 조회수·상담 신청 현황 확인 → 원장 커뮤니티 참여.

### 담임 교사

- 원장에게 반 배정 → 반 미니홈피 활성화(원하는 기능 모듈 ON/OFF) → 반 커버 사진·테마·한 줄 소개 설정 → 오늘의 반 일지 작성 → 사진 업로드 → 학부모 방명록 확인·답글 → 반 공지 발행 → 통신문 연동 발송.

## 5. 비교·신뢰 신호 설계 (핵심 원칙)

원장이 악성 민원에 민감한 시장이므로, 원을 점수로 줄세우는 어떤 장치도 두지 않는다.

### 절대 원칙

- 모든 신호는 "사실의 유무(있다/없다)"만 표시한다.
- 금지: 별점, 등급(A/B/C), 퍼센트, 점수, 평균, 응답률, 순위, 그리고 이 신호로 원을 정렬하는 것.
- 원 A와 원 B를 품질로 비교·서열화할 수 있게 만드는 건 무엇이든 "뒷문 평점"이므로 전부 배제한다.
- 검색 기본 정렬은 거리순 등 중립 기준. "정보 많은 순", "응답 빠른 순" 같은 품질 정렬 금지.
- 공개 후기는 MVP 제외.

### (1) 검증 배지 — 사실 유무

- 정식 인가 시설, 정부 평가인증 시설(여부만, 등급 노출·비교 안 함), CCTV, 식단 공개, 통학 차량, 시간 연장 보육 등.
- 모두 있다/없다로만 표시.

### (2) 투명 운영 — 부담 없이, 더하기만(무벌점)

- 세팅형(1회 세팅, 가끔 갱신): 운영시간·비용·교사·차량·FAQ 등 항목이 채워지면 사실로 노출. 묶어서 "충실도 점수"를 만들지 않는다.
- 자동형(어차피 하는 행동에서 추출, 추가 노동 0): 통신문·공지 발송 시 "소통 활발" 사실 표시, 정원 갱신 시 "정원 정보 최신". 측정·평균·응답률은 만들지 않는다.
- 선택 보너스(강제 X): 식단 월 단위 공개, 사진 최근 공개. 안 해도 불이익 없음.
- 응답성: 공개 비교 지표에서 제외. 원장 본인 대시보드의 사적 지표로만.

### (3) 적합도 태그 — 줄세우기 대신 매칭

- 숲 체험, 영어, 유기농 급식, 소규모, 시간 연장 등 특성 태그. 품질이 아니라 성격.
- 학부모는 "최고"를 고르는 게 아니라 "우리한테 맞는 조건"으로 거른다(필터).

## 6. 통신문·초대장 시스템

원이야의 강점(인터랙티브 페이지)을 통신문에 적용한다. 흔한 공문을 초대장 같은 작품으로 만든다.

### 유형별 템플릿

- 초대장형(행사·감성): 발표회, 재롱잔치, 생일, 소풍, 입학·졸업. 풀 비주얼.
- 안내형(사무): 식단표, 차량 노선 변경, 휴원 안내. 깔끔한 가독성 우선.
- 동의서형(법적): 현장학습 동의, 개인정보 동의. 톤은 입히되 동의 항목은 또렷하게.

### 설계 원칙

- 원장 부담 0: 사진·문구·날짜·장소만 입력하면 템플릿이 자동 완성. 매번 디자인 금지.
- 답장도 같은 형식, 단 한 탭: 참석/불참·동의/비동의·간단 응답을 카드 안에서 한 번에.
- 자동 집계 + 기록: 응답이 원장 콘솔에 자동 집계되고, 열람·동의·응답 시각이 타임스탬프로 기록된다.
- 속은 그대로 증빙: 초대장이어도 법적 기록(열람·동의)은 유지 → 악성 민원 방어 연결.
- 접근성: 날짜·장소·해야 할 것 등 핵심 정보는 항상 상단에 또렷하게. 애니메이션은 거들 뿐.
- 공유: 예쁜 초대장은 학부모가 가족과 공유 → 원이야 자연 노출(마케팅 훅).

## 8. 기술 아키텍처

### 권장 스택

- 프론트엔드: HTML/CSS/JS (현재 운영 중). 향후 Next.js(App Router) on Vercel 전환 검토.
- 백엔드: Supabase (PostgreSQL + Auth + Storage + RLS + Edge Functions).
- 저장소: Supabase Storage(원 사진·앨범·초대장 이미지) — 비공개 버킷 + 만료형 서명 URL.
- 지도: 카카오맵 JavaScript SDK (카카오 개발자 앱 등록 완료, Vercel 환경변수로 키 관리).
- 결제: Toss Payments 정기결제(빌링키) 기반 월 구독.
- 배포: Vercel(프론트), Supabase(백엔드 매니지드).
- 멀티테넌시: 원별 서브경로/서브도메인(예: woniya.co.kr/원이름), 추후 독자 도메인 업셀.

### 역할 분리

- 공개(비로그인): 검색, 원 홈페이지, 배지·태그, FAQ 챗봇, 상담·대기 신청, 공식 문의 작성.
- 학부모(로그인): 통신문 수신·응답, 내 신청 내역, 본인 아이 소속 반 미니홈피 열람·방명록 작성.
- 교사(로그인): 담당 반 미니홈피 운영(일지·사진·공지·방명록 답글·모듈 설정). 서버 권한으로만 접근.
- 원장(로그인): 콘솔(편집·정원·상담·문의·통신문·커뮤니티), 반 생성·교사 배정. 서버 권한으로만 접근.

- 선납 인센티브(옵션): 6개월 선납 시 한 달 추가 또는 프리미엄 템플릿.

- 구독 상태·다음 결제일·초기 약정 만료를 subscriptions 테이블로 관리.

## 16. 반 미니홈피

원이야의 "감성 있는 인터랙티브 페이지" 철학을 반 단위로 확장한 비공개 공간. 해당 반 학부모만 입장 가능하며, 담임 선생님이 운영 주체다.

### 기획 배경

- 원 홈피가 "원을 소개하는 공간"이라면, 반 미니홈피는 "우리 아이 반의 일상을 공유하는 공간"이다.
- 사진·일지·방명록이 쌓일수록 학부모의 플랫폼 재방문이 늘고, 이탈 방어 효과가 강해진다.
- 싸이월드의 "내 미니홈피" 감성을 차용하되, 아이 사진이 포함된 공간이므로 보안을 최우선으로 설계한다.

### 운영 주체 및 권한

- 담임 선생님이 반 미니홈피의 오너다.
- 원장 콘솔에서 반을 생성하고 담임 교사를 배정하면, 해당 교사 계정에 반 관리 권한이 자동 부여된다.
- 원장은 모든 반의 미니홈피를 열람할 수 있으나, 게시물 작성은 담임 교사만 가능하다.
- 학부모는 본인 아이가 속한 반에만 입장할 수 있으며, 방명록 작성만 가능하다.

### 모듈 선택 구조 (선생님이 ON/OFF)

- 오늘의 반 일지 (다이어리)
- 사진 앨범
- 방명록
- 공지사항
- 통신문 연동

### 감성 요소 (싸이월드 레퍼런스)

- 반 커버 사진과 테마 컬러를 선생님이 직접 선택.
- 상단 고정 "선생님의 한마디" 문구.
- 반 소개 한 줄 문구.
- 사진첩은 날짜순으로 자연스럽게 쌓이는 구조.

### 접근 방식

- 비공개형 새 페이지. 로그인 필수, RLS로 접근 제어, noindex 처리.
- 원 홈피 로그인 후 → 내 정보 → 우리 반으로 진입 (메인 경로).
- 선생님이 초대 링크 공유 (보조 경로).

### 보안·개인정보 원칙

- 반 미니홈피 전체는 비공개. 비로그인 접근 및 검색엔진 색인 차단(noindex).
- 사진은 비공개 버킷 + 만료형 서명 URL로만 제공.
- RLS 정책: class_mini, class_posts, class_media, class_comments 테이블은 해당 반 소속 학부모·교사만 SELECT/INSERT 가능.

### 개발 위치

- M5.5. M4(통신문) 완성 후 착수.

## 17. 원 홈페이지 콘텐츠 페이지

노출 방식은 18번(페이지 구조 전략) 참조.

### 필수 콘텐츠 (모달로 노출)

- 오시는 길: 카카오맵 + 교통편 안내 텍스트(주차·대중교통·통학차량). 학부모 상담 방문에 직결.
- 입학안내(모집요강): 모집 대상·정원·기간·절차·준비물·기타 안내. 상담 신청 CTA 앞단 배치.

### 선택 콘텐츠 (원장 ON/OFF, 모달로 노출)

- 소개 영상: 유튜브 등 URL 임베드.
- 원 철학: 원훈·사훈·유아상. 원의 정체성·교육 철학 전달.
- 교육과정: 교육목표·연간교육계획 사실 나열. 비교·순위화 금지.
- 버스 노선: 정류장 지도 + 내 위치 기반 도보 시간. 상세는 19번 참조.

### 별점 금지 원칙 연계

- 모든 콘텐츠 페이지는 "이 원은 이런 것을 한다"는 사실 나열로만 표현한다.
- 품질 강조·비교·순위화 표현 금지. 5번 원칙 적용.

### 데이터 모델 연계

- centers: intro_video_url, directions_text, philosophy(jsonb), pages_enabled(jsonb), has_bus
- admissions: 모집요강 항목
- curriculum: 교육목표·연간교육계획
- bus_routes + bus_stops: 버스 노선 (19번 참조)

## 18. 페이지 구조 전략 — 하이브리드(원페이지 + 모달 + 새 페이지)

### 설계 배경

- 원페이지 스크롤만으로는 콘텐츠가 늘어날수록 한계가 분명하다.
- 모바일 사용자는 새 페이지 이동보다 모달을 선호한다. 뒤로가기 없이 맥락이 유지된다.
- 콘텐츠 성격에 따라 세 가지 방식을 섞는 하이브리드 구조로 간다.

### 3분류 기준

| 분류 | 노출 방식 | 기준 | 해당 콘텐츠 |
|---|---|---|---|
| 원페이지 스크롤 | 메인 페이지에 그대로 | 첫인상·감성·전환 핵심부 | 히어로 배너, 하루 스토리, 대표 사진, 짧은 인사말, 상담 CTA |
| 모달 | 같은 페이지 위 팝업 | 단순 정보 전달 (하위 메뉴 없음) | 입학안내, 오시는 길, 원 철학, 교육과정, 버스 노선, FAQ, 공지, 행사 일정 |
| 새 페이지 | 독립 영역 | 하위 메뉴가 있거나 비공개 보안이 필요한 경우 | 반 미니홈피(비공개·noindex), 버스 노선(지도 상호작용 복잡도 따라 판단) |

### 모달 vs 새 페이지 판단 기준

- 콘텐츠 안에 또 다른 메뉴·탭·하위 구조가 있으면 → 새 페이지
- 단순 정보 나열이면 → 모달
- 비공개 보안이 필요하면 → 새 페이지(noindex + 로그인 필수)
- SEO가 필요한 시점이 오면 → 모달에서 새 페이지로 전환 검토 (현재는 모달 우선)

### 구현 방식

- 모달: 메인 페이지 내 JS로 처리. URL 해시(예: #admissions)로 직접 진입·공유 가능하게.
- 새 페이지(공개형): 공통 HTML + slug 쿼리 + data-theme 상속. 템플릿별 파일 분리 안 함.
- 새 페이지(비공개형): 로그인 필수, RLS 접근 제어, noindex 처리.

### 메뉴·네비게이션 연계

- 모든 템플릿 메뉴(구름/풍선/나뭇잎/액자 버튼 + 햄버거 메뉴)에 콘텐츠 진입점 노출.
- 모달 진입: 클릭 시 팝업.
- 새 페이지 진입: 클릭 시 페이지 이동.

## 19. 버스 노선 페이지

### 기획 배경

- 통학 차량을 운영하는 원은 학부모가 "우리 집 근처까지 버스가 오는지"를 가장 먼저 확인한다.
- 실시간 GPS 추적은 기사 폰 앱 필요·개인정보 이슈·키즈노트 영역 충돌로 범위 밖.
- 정적 정보(노선·정류장·시간표)와 학부모 위치 기반 도보 시간 계산으로 충분한 가치를 제공한다.

### 기능 범위

- 정류장 목록 + 지도 마커 표시 (카카오맵 JavaScript SDK)
- "내 위치로 확인" 버튼 → 브라우저 GPS → 가장 가까운 정류장 + 도보 예상 시간 표시
- 상담 신청 CTA로 자연스럽게 연결
- 버스 운영 원만 메뉴에 항목 노출 (has_bus = true인 경우만)

### 노출 방식

- 기본: 모달 (정보 나열 수준이면)
- 정류장이 많고 지도 상호작용이 복잡하면: 새 페이지(`bus.html?slug={slug}`)로 전환

### 실시간 관련 설계 원칙

- 버스 운행 상태(지연·결행 등)는 제공하지 않는다. 틀린 정보가 아무 정보보다 나쁘다.
- 지연·결행 발생 시 원장이 통신문(공지)으로 즉시 알림 발송하는 흐름으로 연결.

### 데이터 모델

- bus_routes: id, center_id, route_name, operating_time, created_at
- bus_stops: id, route_id, name, address, lat, lng, arrival_time, sort_order

### 온보딩 데이터 수집 방식 (플로우 3: 하이브리드)

- 초기 세팅: 원장이 온보딩 수집 폼에 정류장 명칭·순서·시간 제출 → 드림파파가 카카오맵으로 좌표 확인 후 DB 입력 (1회)
- 이후 수정: 원장이 어드민에서 직접 정류장 추가·수정 (주소 입력 → 카카오 Geocoder 자동 좌표 변환)

### 개발 위치

- M2.5. 카카오 비즈 앱 심사 완료 후 착수.

## 20. 온보딩 정보 수집 폼

### 기획 배경

- 현재 register.html은 가입 신청만 받는다. 실제 원 홈페이지를 세팅하려면 훨씬 많은 정보·이미지가 필요하다.
- 템플릿 4개 설계가 완료되면 각 템플릿에 필요한 정보가 확정된다. 그때 수집 폼을 제대로 설계한다.
- 이 폼이 드림파파 온보딩 작업의 체크리스트가 된다.

### 수집 항목 (템플릿 공통)

- 원 기본 정보: 이름, 주소, 연락처, 운영 시간
- 템플릿 선택
- 원장 인사말 + 원장 사진
- 교사 정보 (이름, 역할, 사진)
- 원 소개 사진 (히어로, 하루 스토리, 앨범)
- 입학안내 정보 (모집 대상, 정원, 기간, 절차, 준비물)
- 오시는 길 (주소 + 교통편 텍스트)
- 버스 운영 여부 → 운영 시 정류장 명칭·순서·시간
- 선택 콘텐츠 ON/OFF (소개 영상 URL, 원 철학, 교육과정)

### 구현 방식

- register.html 확장 또는 별도 onboarding.html 신설
- 단계별 폼 (Step 1: 기본 정보 → Step 2: 사진·미디어 → Step 3: 콘텐츠 페이지 → Step 4: 선택 항목)
- 제출 시 center_applications 테이블에 저장 → 드림파파(super-admin.html)에서 확인 후 세팅 진행

### 개발 위치

- M1.5 (템플릿 완성 직후). 템플릿 4개 히어로 배너·메뉴 버튼이 완성돼야 필요 항목이 확정된다.

## 21. 원장 어드민 콘솔 & 대시보드

### 기획 배경

- 드림파파가 초기 1회 세팅 후 원장에게 콘솔을 인도하는 모델이므로, 원장이 직접 수정할 수 있는 직관적인 편집 UI가 필수다.
- 대시보드는 "원이야가 우리 원에 얼마나 도움이 되는지"를 원장이 직접 확인하는 공간이다. 구독료 5만원을 정당화하는 핵심 근거.

### 어드민 콘솔 편집 항목

- 원 기본 정보: 이름, 주소, 연락처, 운영 시간, 소개 문구
- 사진 관리: 히어로 사진, 하루 스토리 사진, 앨범 사진, 원장·교사 사진
- 정원 상태: 반별 정원·현원·대기 수 갱신
- 콘텐츠 ON/OFF: 소개 영상, 원 철학, 교육과정, 버스 노선 각각 토글
- 입학안내 수정: 모집 대상, 기간, 절차, 준비물 직접 편집
- 오시는 길 수정: 교통편 텍스트 직접 편집
- 버스 노선 수정: 정류장 추가·삭제·시간 변경 (카카오 Geocoder 자동 좌표 변환)

### 편집 UI 설계 원칙

- 모바일에서도 쉽게 수정 가능한 구조 (원장이 폰으로도 빠르게 갱신)
- PC에서는 사이드 내비 + 넓은 본문 레이아웃
- "저장" 즉시 라이브 반영 (미리보기 링크 제공)
- 복잡한 설정은 드림파파에게 요청하는 안내 문구 제공

### 대시보드 수치 (원장 사적 지표)

아래 수치는 원장 본인만 볼 수 있다. 원 간 비교·공개 정렬에 절대 사용하지 않는다. (5번 원칙 적용)

- 홈페이지 조회수: 일간·주간·월간
- 상담 신청 건수: 이번 달 신규 신청 N건 → 입소 확정 N건
- 정원 현황: 반별 현원·대기·여유 한눈에
- 통신문 열람률: 발송 N명 중 열람 N명 (본인 원 내부 지표, 타 원과 비교 안 함)

### 개발 위치

- M2: 기본 편집 UI (원 정보·사진·정원)
- M2.5: 대시보드 + 버스 노선 편집
- M4 이후: 통신문 열람률 대시보드 추가
